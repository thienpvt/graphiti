## execute_cli result_dir ownership (HARD trap — do not mis-implement)
run_live_canary currently rejects any existing result_dir (`result_directory_used`). Therefore execute_cli MUST NOT pre-create output_dir then hand unchanged args to run_live_canary on the success path — that would fail a green canary with result_directory_used.

**Preferred minimal strategy (locked for Task 2):**
1. When `--output-dir` present: validate path is fresh/unused/non-protected (via validate_result_directory or equivalent) BEFORE fallible preflight/Gate0 attestors. Do NOT mkdir yet on the success path.
2. On pretransport failure (manifest / source / execution): atomically create the validated path only inside the failure catch, then call shared early-terminal helper to write the durable trio; never open streamable_http_client.
3. On Gate0 success: leave output_dir absent on disk; open transport; call run_live_canary with the same output_dir arg so run_live_canary owns mkdir + normal artifact lifecycle (post-Gate0 mkdir remains its responsibility unless it also adopts the same create-on-failure pattern for its own early try path).
4. External pre-existing / protected / symlink / used dirs always reject fail-closed; no overwrite; no public CLI/schema flag; no private ownership param required under preferred strategy.
5. Alternative (only if preferred cannot work): private exact-empty prepared-dir ownership parameter with exact-empty proof — not a public CLI/schema flag. Prefer the create-only-on-failure path.

**Success-path regression (mandatory RED/GREEN):** execute_cli with valid fresh output_dir + successful Gate0 attestors reaches streamable_http_client / run_live_canary without result_directory_used; artifacts still persist normally under run_live_canary ownership.
